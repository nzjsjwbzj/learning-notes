# 6.编码

![image-20250618111308196](./%E8%AE%A1%E7%BD%91.assets/image-20250618111308196.png)

脉冲编码调制（PCM）是一种将模拟信号数字化的方法，它通过采样、量化和编码三个步骤来实现语音信号的数字化。采样是对模拟信号进行周期性扫描，将其转换为一系列离散的样本；量化是将这些样本的幅度值进行离散化处理；编码则是将量化后的值用二进制数来表示。这样，语音信号就被转换成了计算机可以处理的数字形式。

幅度调制是一种模拟调制技术，它通过改变载波信号的幅度来传输信息，并不涉及信号的数字化过程。曼彻斯特编码和差分曼彻斯特编码则是用于数据链路层的编码技术，它们主要用于同步数据传输和错误检测，而不是语音信号的数字化。

曼彻斯特编码是一种**物理层数据传输编码技术**，核心特点是通过**电平跳变**同时表示数据和时钟信息,**比特表示规则**：

- **比特0**：电平从 **高→低 跳变**（或定义为 **上升沿**）
- **比特1**：电平从 **低→高 跳变**（或定义为 **下降沿**）

# 7.奈奎斯特定理

![image-20250618114846494](./%E8%AE%A1%E7%BD%91.assets/image-20250618114846494.png)

根据奈奎斯特定理，在无噪声情况下，通信链路的最大数据传输速率（C）可以通过以下公式计算：

C = 2 × H × log2N

其中，H是信道的带宽，N是一个码元所取的离散值个数。题目中给出的带宽（H）为3kHz，采用4个相位，每个相位具有4种振幅的QAM调制技术，这意味着N = 16（即16种符号状态）。

将这些值代入公式中：

C = 2 × 3kHz × log216

计算结果为24kbps。因此，该通信链路的最大数据传输速率是24kbps

# 9.曼彻斯特

![image-20250618115518436](./%E8%AE%A1%E7%BD%91.assets/image-20250618115518436.png)

曼彻斯特编码是一种将时钟信号与数据信号同步编码的技术。其主要特点如下：

- 每个码元被分成两个相等的间隔，分别用高电平和低电平表示不同的数据值。例如，前一个间隔为高电平而后一个间隔为低电平表示码元1，反之则表示码元0。
- **位中间的跳变用于时钟信号，而每个码元的电平表示数据信号**。
- 曼彻斯特编码将时钟与数据包含在信号流中，在传输代码信息的同时，也将时钟同步信号一起传输到接收方。
- 这种编码机制适合传输二进制数字信号，而非模拟信号如声音。

选项A：每个信号起始边界作为时钟信号有利于同步。

- 错误。曼彻斯特编码是通过位中间的跳变来作为时钟信号，而不是信号的起始边界。

选项B：将时钟与数据取值都包含在信号中。

- 正确。曼彻斯特编码的特点之一就是将时钟信号与数据信号同时包含在信号流中。

选项C：这种模拟信号的编码机制特别适合于传输声音。

- 错误。曼彻斯特编码适合传输二进制数字信号，而非模拟信号如声音。

选项D：每位的中间不跳变表示信号的取值为0。

- 错误。曼彻斯特编码规定，位中间的跳变用于时钟信号，而不是直接表示数据的取值。

经过分析，只有选项B符合曼彻斯特编码的特点。



# 10.PPP协议

![image-20250618121532174](./%E8%AE%A1%E7%BD%91.assets/image-20250618121532174.png)

在同步传输中，PPP（Point-to-Point Protocol）协议采用零比特填充法来实现透明传输。这意味着在发送数据之前，PPP会扫描整个数据帧，一旦发现五个连续的“1”，就会立即在后面插入一个“0”，**从而确保数据帧中不会出现连续的六个“1”**，这与帧定界符的模式相区别。

对于给定的比特串0111110001111110，应用零比特填充法后，应在第二个连续的五个“1”之后插入一个“0”。因此，组帧后的比特串应该是01111100011111010。

# 15.子网掩码

![image-20250618130737371](./%E8%AE%A1%E7%BD%91.assets/image-20250618130737371.png)

### 如何查看和理解子网掩码：

1. **格式**：

   - 子网掩码通常由连续的1和0组成，1表示网络位，0表示主机位。
   - 常见的表示形式有：
     - **点分十进制表示法**：例如 `255.255.255.0`。
     - **前缀长度表示法（CIDR）**：例如 `/24`，表示前24位是网络位。

2. **常见的子网掩码示例**：

   - `255.0.0.0` → `/8`（A类默认掩码）
   - `255.255.0.0` → `/16`（B类默认掩码）
   - `255.255.255.0` → `/24`（C类默认掩码）
   - `255.255.255.128` → `/25`（将C类网段划分为2个子网）
   - `255.255.255.192` → `/26`（划分为4个子网）

3. **如何计算网络地址和广播地址**：

   - **网络地址**：将IP地址与子网掩码进行**按位与运算**。
   - **广播地址**：将网络地址的主机位全部置1。

   **示例**：

   - IP地址：`192.168.1.100`

   - 子网掩码：

     ```
     255.255.255.0
     ```

     （即

      

     ```
     /24
     ```

     ）

     - 网络地址：`192.168.1.0`
     - 广播地址：`192.168.1.255`

4. **子网掩码的判断方法**：

   - 观察子网掩码中1的连续长度（从左到右，不能间断）。
   - 例如：
     - `255.255.255.0` 的二进制是 `11111111.11111111.11111111.00000000`，所以网络位是24位。
     - `255.255.255.128` 的二进制是 `11111111.11111111.11111111.10000000`，所以网络位是25位。

5. **实用场景举例**：

   - **判断两个IP是否在同一子网**：
     将两个IP地址分别与子网掩码做按位与运算，如果结果相同，则在同一子网。
     - IP1：`192.168.1.100`，IP2：`192.168.1.200`，子网掩码：`255.255.255.0`
     - 网络地址都是 `192.168.1.0` → 在同一子网。
   - **计算可用主机数量**：
     主机位数量 = 32 - 网络位长度（前缀长度）
     可用地址数 = 2主机位数2主机位数 - 2（减去网络地址和广播地址）
     - 例如 `/24` 的子网：主机位=8，可用地址数=2828 - 2 = 254。

# 17.UDP，IP数据报

![image-20250618132625630](./%E8%AE%A1%E7%BD%91.assets/image-20250618132625630.png)

当一个UDP用户数据报的数据字段为8192字节，并且在链路层使用以太网进行传送时，我们需要考虑以太网的MTU（最大传输单元）限制。**标准以太网的MTU为1500字节，这包括了IP首部和UDP数据报的数据字段**。IP首部固定长度为20字节，UDP首部长度为8字节，因此可用于数据字段的最大长度为1500 - 20 - 8 = 1472字节。

由于UDP数据报的总长度为8192字节 + 8字节（UDP首部）= 8200字节，超过了单个以太网帧的承载能力，因此需要将UDP数据报划分为多个IP数据报片来进行传输。

划分过程如下：

1. 第一个数据报片包含1472字节的数据（因为1472 + 20 + 8 = 1500，符合MTU限制）。
2. 后续每个数据报片（直到倒数第二个）也将包含1472字节的数据。
3. 最后一个数据报片将包含剩余的数据，即8200 - (1472 × 5) = 800字节。







> **UDP头部只存在于第一个IP数据报片中，其它分片中不会重复包含UDP头部。**

------

## 🔍 原因解析：

### 1. **分片是在 IP 层完成的**

- UDP 是运输层协议，它会先把数据和 UDP 首部组合成一个**完整的 UDP 数据报**（总长度为 8200 字节）。
- 然后这个 UDP 数据报被交给 IP 层，IP 层再根据 MTU 进行分片。

### 2. **分片的是 IP 数据报，不是 UDP 数据报**

- IP 层收到的是一个完整的 UDP 数据报（包括 UDP 头部）。
- 分片之后，**IP 每个片段都只有一个 IP 头**（20 字节），而：
  - **第一个分片还会保留 UDP 头（8 字节）**；
  - **其余分片只包含数据部分，没有 UDP 头**。

------

## 🧠 举个例子来说明：

你要传输的 UDP 数据报是 8200 字节：

- 包括：8 字节 UDP 头 + 8192 字节 UDP 数据。

**MTU 限制为 1500 字节，因此每个 IP 数据报片最大只能携带 1480 字节（1500 - 20）。**

### 分片情况如下：

| 分片编号 | IP头部 | UDP头部 | 有效负载（数据） | 总长度（字节） | 备注      |
| -------- | ------ | ------- | ---------------- | -------------- | --------- |
| 第1片    | 20字节 | 8字节   | 1472字节         | 1500           | 包含UDP头 |
| 第2片    | 20字节 | ×       | 1480字节         | 1500           | 无UDP头   |
| 第3片    | 20字节 | ×       | 1480字节         | 1500           | 无UDP头   |
| 第4片    | 20字节 | ×       | 1480字节         | 1500           | 无UDP头   |
| 第5片    | 20字节 | ×       | 1480字节         | 1500           | 无UDP头   |
| 第6片    | 20字节 | ×       | 808字节          | 828            | 无UDP头   |



注意最后一片的数据长度是 808，因为：
 8200（总） - 8（UDP头） - 1472×5 = **808**

# 20.域名

![image-20250618141407501](./%E8%AE%A1%E7%BD%91.assets/image-20250618141407501.png)



域名

- **本质**：互联网上**人类可读的地址标识**（如 `google.com`）。
- **核心价值**：将难记的IP地址（如 `142.250.179.14`）转换为易记的字符组合。
- **类比**： IP地址 = 地理坐标（纬度/经度） 域名 = 街道名称（如“北京市海淀区中关村大街”）

**域名服务器（DNS Server）**

核心功能**

将**域名翻译为IP地址**（正向解析），或将**IP地址反查域名**（反向解析），是互联网的“翻译官”和“导航系统”。



##  什么是域名服务器（DNS Server）？

域名系统（DNS）负责把我们熟悉的“人类友好”地址（如 `www.baidu.com`）转化为计算机能识别的 IP 地址（如 `220.181.38.251`）。这个过程不是一下子就知道的，而是**一层层向上查询**，这就是域名服务器的层次结构。

------

## 🧭 DNS 层次结构 = 一棵倒挂的树 🌲

```
yaml复制编辑        根（Root）
          |
    ┌─────┼─────┐
   com   cn   org  ...（顶级域）
    |     |
 baidu  edu  ...（二级域）
    |
www（主机名/子域名）
```

------

## 🧱 域名服务器的层级结构分为以下几类：

### 1. **根域名服务器（Root DNS）**

- 顶层，全球只有13组（分布在多个国家）。
- 作用：告诉你**某个顶级域（如`.com`、`.cn`）的服务器在哪**。

------

### 2. **顶级域名服务器（TLD DNS）**

- 管理 `.com`、`.cn`、`.org` 等。
- 比如你查 `www.baidu.com`，TLD DNS 会告诉你：“`.com`下有个叫 `baidu` 的域，它的服务器在这里”。

------

### 3. **权威域名服务器（Authoritative DNS）**

- 是**最终知道你要查的域名的 IP 地址的服务器**。
- 比如 `www.baidu.com` 的 IP 地址就保存在 Baidu 的权威 DNS 服务器上。
- 它可以直接回答：“这个域名对应的 IP 地址是 220.181.38.251”。

------

### 4. **本地（递归）域名服务器（Local/Recursive DNS）**

- 你家或公司网络运营商提供的 DNS。
- 浏览器一查地址，它就先来问这个服务器。
- 如果它不知道，它就去问根 → TLD → 权威，最终拿到 IP 再返回给你，并缓存结果。

------

## 📦 查询流程举例（查 `www.baidu.com`）：

1. 你的电脑 → 本地DNS：问“www.baidu.com是啥？”
2. 本地DNS不认识 → 去问 **根DNS**：“.com在哪？”
3. 根DNS告诉它：`.com`的服务器在这儿。
4. 本地DNS → `.com` TLD DNS：请问 `baidu.com`在哪？
5. `.com` DNS → 回答：`baidu.com`的权威服务器是这个地址。
6. 本地DNS → 权威DNS：请问 `www.baidu.com` 对应什么 IP？
7. 权威DNS：IP 是 220.181.38.251
8. 本地DNS缓存 → 返回 IP 给你 → 浏览器连接该 IP

------

## ✅ 总结一句话：

> **DNS 查询就像一层层打电话问路：从根服务器开始，一直找到真正“知道答案的权威服务器”。这就是域名服务器的层次结构。**

# 22.Internet和internet

![image-20250618142740154](./%E8%AE%A1%E7%BD%91.assets/image-20250618142740154.png)

在本题中，我们需要判断哪一个选项对相关术语的解析是错误的。以下是对各个选项的详细分析：

1. **Internet**：指的是全球范围内互联的计算机网络系统，通常被称为“互联网”。它由许多较小的网络互联而成，是一个全球性的、开放的、由许多互连的计算机网络组成的系统。
2. **internet**：指的是将多个计算机网络互联成大型网络系统的技术，这个术语是通用的，泛指任何一组互联的网络，无论规模大小。
3. **Intranet**：指的是在企业或组织内部使用的局域网（LAN）或广域网（WAN），通常不直接连接到Internet，但可以通过防火墙和代理服务器等设备进行连接。

# 23.Internet系统

![image-20250618143736403](./%E8%AE%A1%E7%BD%91.assets/image-20250618143736403.png)

在分析关于Internet核心交换与边缘部分的描述时，我们需要理解以下几点：

1. **Internet的组成**：
   - **边缘部分**：**由所有连接到Internet的主机组成**，这些主机可以是用户的终端设备，如计算机、手机等。
   - **核心部分**：**由大量的路由器和网络组成**，这些路由器和网络负责数据包的转发和传输。
2. **核心交换部分的功能**：
   - 核心交换部分主要为数据包的传输提供服务，确保数据能够从源端系统传输到目的端系统。
   - 它不直接与应用程序进程进行通信，而是提供基础的网络连接服务。
3. **应用程序运行的位置**：
   - 网络应用程序通常运行在边缘部分的端系统上，而不是核心交换部分。

# 总结香奈

奈奎斯特定理（Nyquist's Theorem）：

- 该定理描述了在*无噪声*的理想信道中，最大数据传输速率（即码元速率）与信道带宽的关系。
- 公式为：最大数据传输速率 = 2 × 带宽（单位为Hz）× log2(码元状态数)（单位：比特/秒）。
- 注意：码元状态数是指调制中使用的不同信号状态的数量。例如，二进制调制（两个状态）时，log2(2)=1，所以最大数据速率=2×带宽×1。
- 奈奎斯特定理给出了无噪声信道下的理论极限，但实际上信道总是有噪声的。

香农公式（Shannon's Formula）：

- 香农公式（香农定理）则考虑了*有噪声*的信道，给出了有噪声信道的最大信息传输速率（即信道容量）的上限。
- 公式为：信道容量（单位：比特/秒）= 带宽（单位为Hz）× log2(1 + SNR)。
- 其中SNR是信噪比（信号功率与噪声功率的比值），通常用线性值（不是分贝）代入公式。
- 香农公式给出了在任意小的错误率下，通过有噪声信道可靠传输的最大速率，也称为香农极限。

两者的区别：

1. 奈奎斯特定理假设无噪声，而香农公式考虑了噪声。
2. 奈奎斯特定理强调码元状态数对速率的影响，而香农公式强调信噪比对速率的影响。
3. 实际系统中，最大传输速率受这两个定理共同限制，即不能超过奈奎斯特速率（无噪声时）和香农容量（有噪声时）中的最小值。

举例说明：
假设一个无噪声信道，带宽为3kHz，采用8相调制（即每个码元有8个状态），则根据奈奎斯特定理：
最大数据速率 = 2 × 3000 × log2(8) = 2×3000×3 = 18 kbps。

假设同一个信道有噪声，信噪比SNR=100（线性值，相当于20dB），则根据香农公式：
信道容量 = 3000 × log2(1+100) ≈ 3000 × log2(101) ≈ 3000 × 6.66 ≈ 20 kbps。
因此，在该有噪声信道下，无论采用何种调制方式，实际传输速率不可能超过20kbps，并且由于使用了8相调制，奈奎斯特极限为18kbps，所以最终最大速率不会超过18kbps（但可能低于香农极限）。

总结：
奈奎斯特定理：无噪声信道中，最大数据速率由带宽和码元状态数决定。
香农公式：有噪声信道中，信道容量（最大可靠传输速率）由带宽和信噪比决定。

# 39.ARQ协议

![image-20250618170212171](./%E8%AE%A1%E7%BD%91.assets/image-20250618170212171.png)

在连续自动重传请求（ARQ）协议中，发送方可以在不等待确认的情况下连续发送多个数据帧。接收方在收到每个数据帧后都会发送确认帧（ACK），以告知发送方该帧已被正确接收。如果接收方在接收到某个数据帧时检测到错误，它将丢弃该帧，并且不会发送该帧的确认帧。

假设发送方连续发送了前5个数据帧，接收方在接收到第4个数据帧时检测到了错误，那么接收方只会对前3个正确的数据帧发送确认帧。因此，接收方最后发送的确认帧的序号将是第3个数据帧的确认序号。



# 40.PPP协议

![image-20250618173402388](./%E8%AE%A1%E7%BD%91.assets/image-20250618173402388.png)

在数据链路层协议PPP（Point-to-Point Protocol，点对点协议）的协商过程中，包含了多个阶段，每个阶段都有特定的功能和目的。首先，PPP协议的协商过程包括链路建立阶段、认证阶段、网络层协议阶段和链路终止阶段。在认证阶段，如果认证失败，PPP协议会进入链路终止阶段。认证阶段的目的是确认通信双方的身份，确保数据传输的安全性。如果认证成功，则进入网络层协议阶段

A. **Network**：这是PPP链路建立后的状态，表示网络层协议已经开始运行。如果身份认证失败，链路不会直接进入此状态。
B. **Authenticate**：这是进行身份认证的状态。如果认证失败，链路不会停留在该状态，而是会进入下一个状态。
C. **Terminate**：当身份认证失败时，PPP链路会终止，这意味着通信双方将断开连接，释放资源。在这种情况下，C选项是正确的。
D. **Dead**：这是PPP链路的初始状态，表示链路尚未建立或已经断开。如果身份认证失败，链路不会回到此状态

## 1. 以太网（Ethernet）是什么？

以太网就是我们平常说的“网线网络”，是目前最常用的**局域网通信方式**。

### ✅ 特点：

- 使用网线（如 RJ45）连接电脑、路由器、交换机。
- 按照一定规则发送/接收数据包。
- 每台设备都有一个 **MAC地址**，用来标识“谁是谁”。

### ✅ 举个例子：

假如办公室里有几台电脑，它们通过网线连接在交换机上，这样大家就能互相发消息、共享文件——这个网络就叫“以太网”。

## 🔄 2. PPP 协议（Point-to-Point Protocol）

PPP 是一个 **点对点协议**，常用于通过电话线、串口、光猫等建立 **一对一的数据连接**。

### ✅ 它主要做两件事：

- **建立连接**（谁对谁连上了）
- **传输数据**（保证数据稳定、安全地发送）

### ✅ 应用场景：

- 拨号上网（早期用猫拨号）
- 串口通信
- 通过 USB 模拟网络（比如 4G 模块）

### ✅ 类比理解：

PPP 就像一根“数据电话线”，两台设备之间“一对一通话”，它不管你走哪条物理线路，只管“把数据包安全送过去”。

## 🌉 3. PPPoE 协议（PPP over Ethernet）

PPPoE 就是 **把 PPP 协议放到以太网中来用**。

### ✅ 为什么要这样做？

原本 PPP 是用在“电话线/串口”上的，但后来网络走网线（以太网）了。运营商又不想改 PPP 认证机制，就发明了 PPPoE。

### ✅ 应用场景：

- 家里装宽带，光猫拨号，或者你电脑拨号上网。
- 运营商通过 PPPoE 来 **识别你是谁、计费、分配IP地址**。

### ✅ 类比理解：

假设 PPP 是“打电话通信”，以太网是“高速公路”，那 PPPoE 就是“在高速公路上跑的电话车”。

## ✅ 总结对比：

| 名称   | 用途           | 特点                         |
| ------ | -------------- | ---------------------------- |
| 以太网 | 局域网传输数据 | 快速、支持多台设备同时通信   |
| PPP    | 一对一通信协议 | 串口/拨号通信、点对点        |
| PPPoE  | 以太网上的 PPP | 用于宽带拨号、带账号密码认证 |

# 56.帧中继

![image-20250619150616316](./%E8%AE%A1%E7%BD%91.assets/image-20250619150616316.png)

backward explicit congestion notification

帧中继网（**Frame Relay Network**）是 1980~90 年代广泛使用的一种**广域网（WAN）技术**，它用于在多个地点之间快速传输数据，是在 X.25 之后出现的更快、更高效的一种网络。

| 项目     | X.25                | 帧中继                       |
| -------- | ------------------- | ---------------------------- |
| 检错重发 | 每一跳都检错 + 重发 | 端到端才管错误，转发节点不管 |
| 速度     | 慢（如 9.6kbps）    | 快（几 Mbps 到几十 Mbps）    |
| 效率     | 低（每层都校验）    | 高（只管转发）               |
| 使用场景 | 电话线、老式通信    | 企业专线、早期广域网         |

![image-20250619145306379](./%E8%AE%A1%E7%BD%91.assets/image-20250619145306379.png)

### 步骤 1：建立逻辑连接（PVC）

帧中继采用 **永久虚电路（PVC，Permanent Virtual Circuit）** 或 **交换虚电路（SVC）**。大多数场景下使用的是 PVC。

**PVC 是提前配置好的逻辑通道**，运营商在后台为你绑定 A ↔ B 一条通路。
 你不用像 TCP 一样每次通信都重新“拨号”。

------

### ✅ 步骤 2：封装数据为帧

A 站点要发送数据，会做以下操作：

- 把数据封装成一个**帧（Frame）**
- 在帧头中加入 **DLCI（Data Link Connection Identifier）编号**

这个 DLCI 编号告诉中继网络：“这个帧该送往哪条虚电路（哪台设备）”

------

### ✅ 步骤 3：转发中继过程

数据帧发送到帧中继网络后，**每个中继节点只看 DLCI 编号，不看内容**：

- 类似快递中心看快递单号，只看“送哪儿”，不拆包裹。
- 中继设备不做重传、不做确认，只负责“尽力转发”。

这就让帧中继的**延迟低、效率高**。

------

### ✅ 步骤 4：到达接收方 B

接收方 B 接收到帧：

- 读取帧内容
- 提取出用户数据（上层协议）
- 没有收到的话，也没人告诉你——由上层协议（如 TCP）去负责错误处理

------

### ✅ 步骤 5：拥塞控制机制

帧中继有简单的**拥塞标志位**：

- FECN（Forward Explicit Congestion Notification）：转发方向拥塞
- BECN（Backward Explicit Congestion Notification）：返回方向拥塞
- DE（Discard Eligibility）：非关键帧，网络忙时可以丢弃

这几位不是强制控制，而是提醒你：“该减速了！”

------

### ✅ 步骤 6：上层协议保障可靠性（如 TCP）

因为帧中继本身不可靠：

- 没有重传
- 没有顺序控制
- 不确认是否送达

所以通常配合**上层协议（比如 TCP）**来实现可靠传输。

# 59.CSMA/CA

![image-20250619151509907](./%E8%AE%A1%E7%BD%91.assets/image-20250619151509907.png)

首先，CSMA/CA协议的核心思想是在发送数据前先侦听信道是否空闲，以避免冲突。当源主机检测到信道空闲后，它并不会立即发送数据，而是会等待一个短的时间间隔（SIFS或DIFS），这是为了确保其他可能也在等待发送的高优先级帧能够优先发送。因此，选项B描述是错误的。

接下来，如果信道在等待间隔后仍然空闲，源主机才会发送数据帧。发送完成后，源主机并不会立即认为发送成功，而是会等待目的主机发送的ACK确认帧。这一点在选项A和C中都有正确描述。

目的主机在正确接收到数据帧后，会在SIFS（短帧间间隔）之后发送ACK确认帧，以告知源主机数据帧已被成功接收。这与选项D的描述相符

# 66.IP

![image-20250619163042569](./%E8%AE%A1%E7%BD%91.assets/image-20250619163042569.png)



### 1. 确定地址类型（①）

IP地址分类由**第一个字节的范围**决定：

- A类：`0~127`
- B类：`128~191`
- C类：`192~223`

题目中IP地址为 `172.33.179.9`，第一个字节 `172` 属于 `128~191` 区间，因此是 **B类IP地址**。

### 2. 计算网络地址（②）

网络地址由 **IP地址与子网掩码按位与（AND）** 得到。

- **IP地址二进制**：`172.33.179.9` → 拆分字节为二进制： `172` → `10101100`；`33` → `00100001`；`179` → `10110011`；`9` → `00001001`。
- **子网掩码二进制**：`255.255.192.0` → 拆分字节为二进制： `255` → `11111111`；`255` → `11111111`；`192` → `11000000`；`0` → `00000000`。
- **按位与运算**：仅保留子网掩码为`1`的位（前18位，因`255.255.192.0`是`/18`子网，前18位为网络位）：
  - 前2字节（`172.33`）：子网掩码为`255.255`，与运算后不变。
  - 第3字节（`179` → `10110011`）：与`192`（`11000000`）按位与 → 结果为`10000000`（即`128`）。
  - 第4字节（`9` → `00001001`）：与`0`按位与 → 结果为`00000000`（即`0`）。

因此，**网络地址**为 `172.33.128.0`。

### 3. 计算直接广播地址（③）

直接广播地址是子网内**主机位全为1**的地址。

- 子网掩码`255.255.192.0`是`/18`，即**前18位为网络位，后14位为主机位**。

- 网络地址为

  ```
  172.33.128.0
  ```

  ，需将主机位（后14位）设为

  ```
  1
  ```

  ：

  - 第3字节：网络位为前2位（`10`），主机位为后6位 → 设为`111111`（即`63`），因此第3字节为 `128 + 63 = 191`。
  - 第4字节：主机位全为`1` → 即`255`。

因此，**直接广播地址**为 `172.33.191.255`。

### 4. 计算主机号（④）

主机号是IP地址中**主机位的部分**（即网络地址以外的位）。

- 网络地址为

  ```
  172.33.128.0
  ```

  ，IP地址为

  ```
  172.33.179.9
  ```

   

  → 主机位对应数值为：

  - 第3字节：`179 - 128 = 51`
  - 第4字节：`9 - 0 = 9`

因此，**主机号**为 `51.9`（点分十进制表示主机位的拆分）。

### 5. 子网内第一个可用IP地址（⑤）

子网内第一个可用IP是**网络地址 + 1**（网络地址不可分配给主机，第一个可用地址为网络地址的下一个地址）。

网络地址为`172.33.128.0`，因此**第一个可用IP地址**为 `172.33.128.1`。

### 选项A验证

选项A提到“①B类IP地址”，结合第一步分析，IP地址`172.33.179.9`的第一个字节`172`属于B类地址范围（`128~191`），因此**选项A正确**。

综上，表格各空的关键结论为：

- ①：B类IP地址（选项A正确）
- ②：`172.33.128.0`
- ③：`172.33.191.255`
- ④：`51.9`
- ⑤：`172.33.128.1`

## 🧩 1. MF（More Fragments）——**是否还有分片**

- **1 表示：后面还有更多分片（没完）**
- **0 表示：这是最后一个分片（结束）**

这是三位 Flags 中的其中一位。

### 🔹举例：

如果一个 IP 数据报被分成 3 片：

| 分片  | MF                |
| ----- | ----------------- |
| 第1片 | 1                 |
| 第2片 | 1                 |
| 第3片 | 0（表示最后一片） |



------

## 🛑 2. DF（Don't Fragment）——**禁止分片**

- **1 表示：禁止分片（Don't Fragment）**
- **0 表示：允许分片**

这是 Flags 的另一位。

### 🔹用途：

某些协议（如 Path MTU Discovery）希望全路径中**不分片**，就设置 DF=1，如果中途发现太大，**直接丢包并发 ICMP 回报错误**。

------

## 🔁 3. 偏移量为什么以 8 字节为单位？

这个字段叫：**Fragment Offset（片偏移）**

它的单位是 **8 个字节（即 64 bit）**，不是字节！

### 🔸 为什么不是字节？

因为 IP 报文的最大长度是 65535 字节，而 Fragment Offset 只有 **13 位（最大值 8191）**，如果单位是字节，不够表示。

所以：

> 用 8 字节作单位，最大就能表示：8191 × 8 = 65528 字节 ≈ 64KB，刚好够！

------

### 🔹举个完整分片例子：

假设原始 IP 数据报为：4000 字节
 MTU = 1500 → 数据每片最多 1480 字节（减去 20 字节 IP 头）

| 分片  | 数据长度 | 偏移量（字节） | 偏移量（字段值） | MF   |
| ----- | -------- | -------------- | ---------------- | ---- |
| 第1片 | 1480     | 0              | 0                | 1    |
| 第2片 | 1480     | 1480           | 1480 ÷ 8 = 185   | 1    |
| 第3片 | 1040     | 2960           | 2960 ÷ 8 = 370   | 0    |



------

# 73.ICMP和TTL Tracert

![image-20250619184557931](./%E8%AE%A1%E7%BD%91.assets/image-20250619184557931.png)

时间超过差错报文

##  一、TTL 字段（Time To Live）

### 📌 TTL 是什么？

TTL 是 IP 数据报头中的一个字段，表示**该数据报还能经过多少个路由器（跳数）**。

| 字段名 | 大小 | 作用                       |
| ------ | ---- | -------------------------- |
| TTL    | 8 位 | 限制数据包在网络中生存时间 |



------

### 🧠 TTL 的工作方式：

- 初始值通常是 `64`、`128` 或 `255`
- 每经过一个路由器，TTL 就减 `1`
- **如果 TTL 减到 0**，该路由器就**丢弃这个数据报**，并触发一个 ICMP 报文通知源主机

------

## 📨 二、ICMP 时间超过（Time Exceeded）报文

### 📌 是什么？

ICMP 是 IP 的“差错报告协议”，当 TTL=0 数据包被丢弃时，**中间路由器**会发送一个 **ICMP Type 11 Code 0 的“时间超过”报文** 回给源主机。

| ICMP 报文名称 | 类型（Type） | 代码（Code） | 说明                     |
| ------------- | ------------ | ------------ | ------------------------ |
| Time Exceeded | 11           | 0            | TTL 到达 0，数据包被丢弃 |



------

### ✈️ 举个例子：

你从电脑发出一个数据包给远方的服务器，设 TTL=3：

1. 第1跳（本地路由器）：TTL=3→2，继续转发
2. 第2跳（运营商设备）：TTL=2→1，继续转发
3. 第3跳：TTL=1→0，**被丢弃**
4. 第3跳发送 ICMP 时间超过 报文返回你的电脑



tracert

`tracert` 通过发送一系列 TTL 从 1 开始递增的数据包，利用每一跳路由器返回的 **ICMP 时间超过报文**，逐跳显示出网络路径。

##  工作原理分步骤讲清楚：

### 步骤 1：发送第一个包，TTL=1

- 第一个路由器收到后，TTL=0，**丢包**
- 它发送一个 **ICMP Time Exceeded（类型11）** 报文给你
- 你就知道：第 1 跳是哪个路由器

------

### 步骤 2：发送第二个包，TTL=2

- 第一个路由器 TTL=1，转发
- 第二个路由器 TTL=0，**丢包 + 发 ICMP 报文**
- 你就知道：第 2 跳是谁

------

### 步骤 3：继续 TTL=3、4、5...

- 每次 TTL 增加 1，直到包成功到达目标主机，或超时
- 最后一跳不再返回 ICMP 超时报文，而是目标主机的正常回应（如 ICMP Echo Reply）

------

## 📦 每一个数据包都有什么？

- TTL：逐步从 1 开始递增
- 协议：一般是 ICMP（ping 请求）或 UDP（Linux traceroute 默认）
- 每跳返回一个 ICMP 报文（类型 11） → 包含**哪台设备丢掉了包**



# 74.RIP协议

RIP（Routing Information Protocol，路由信息协议）是最早的**内部网关路由协议**之一，用来在**小型网络中自动选路径**。

### 📌 一句话解释：

> **RIP 会让路由器之间每隔一段时间互相“广播自己的路由表”**，大家比一比哪条路跳数最少，就走哪条。

| 🔁 周期性更新         | 每隔 30 秒广播一次自己的路由表                             |
| 🔃 路由算法           | 使用 **距离向量算法（Distance Vector）**                   |
| 🛡 环路防止机制       | 有多种机制（如水平分割、毒性逆转、触发更新）               |
| 📮 使用的协议         | 运行在 UDP 端口 520 上                                     |

# 79.BGP协议

![image-20250619192552620](./%E8%AE%A1%E7%BD%91.assets/image-20250619192552620.png)

BGP（边界网关协议）是一种用于在不同自治系统（AS）之间交换路由信息的协议。针对题目中的选项进行分析：

A. BGP确实要求相邻的AS边界路由器之间交换到达目的的网络的路径。这是BGP的基本功能之一，即通过交换网络层可达性信息（NLRI）来告知其他AS如何到达特定的网络前缀。所以，A选项描述是正确的。

B. BGP不仅在路径发生变化时才交换路由信息。BGP路由器会定期发送Keepalive消息来确认连接的有效性，并且在路由信息有更新时也会及时交换。所以，B选项描述是错误的。

C. BGP的路由选择算法确实是基于路径向量算法。它通过维护一个包含到达每个目的网络的所有可用路径的列表来选择最佳路径。所以，C选项描述是正确的。

D. BGP路由器的路由表确实需要包括分组到达目的网络的路径。这些路径信息是通过BGP更新消息交换得到的。所以，D选项描述是正确的。

**BGP（边界网关协议）** 是用来连接全球各个“国家级”网络（自治系统，AS）之间的“外交官协议”，掌控着整个互联网的路由。

# 80.路由表和转发表

![image-20250619193458031](./%E8%AE%A1%E7%BD%91.assets/image-20250619193458031.png)

A. **路由表是根据路由选择算法得出，转发表是从路由表得出**。
\- **描述正确**：路由表是通过路由选择算法（如RIP、OSPF、BGP等）动态生成的，它记录了到达各个目标网络的最优路径。转发表则是从路由表中提取出来的，用于快速转发数据包。转发表通常包含更简洁的信息，如目标网络地址、下一跳地址和输出接口等，以便硬件能够快速地进行数据包转发。

B. **路由器的交换结构可以通过路由选择协议生成路由表**。
\- **描述错误**：路由器的交换结构主要负责数据包的转发，而路由表是由路由选择协议（如RIP、OSPF、BGP等）生成的。这些协议通过交换路由信息来更新和维护路由表。

C. **路由器的转发分组是物理层的主要工作**。
\- **描述错误**：转发分组主要是数据链路层或网络层的工作，而不是物理层。物理层主要负责比特流的传输，而转发分组涉及到数据包的封装、解封装和转发等操作，这些操作主要在数据链路层或网络层进行。

D. **路由器的选择处理机根据转发表对分组从输入端口转发到输出端口**。
\- **描述错误**：路由器的转发操作通常是由硬件（如交换结构）完成的，而不是由选择处理机（CPU）直接处理。选择处理机主要负责路由表的计算和更新，以及一些复杂的控制和管理功能。当数据包到达路由器时，硬件会根据转发表快速地将数据包从输入端口转发到输出端口。

# 84.多播生成树RPB

![image-20250624104239499](./%E8%AE%A1%E7%BD%91.assets/image-20250624104239499.png)

一、RPB的核心原理

1. 反向路径转发（RPF）检查

- 路由器接收到多播数据包时，会检查数据包的源IP是否通过“到达源主机的最短路径”进入接口。若符合，则转发数据包；否则丢弃（避免环路）。

- 例：若路由器A到多播源的最短路径是接口X，当A从接口X收到多播包时，会向其他所有接口转发；若从非X接口收到，则丢弃。

2. 基于生成树的广播

- 以多播源为根，构建一棵覆盖所有网络的“反向路径生成树”，数据包沿树的分支转发，确保每个网段仅接收一次，避免重复传输。



特点：没有环路，每个节点只接收一次

# 87.运输层

![image-20250624115947613](./%E8%AE%A1%E7%BD%91.assets/image-20250624115947613.png)

通信的端点并不是主机，而是主机中的应用进程。运输层通过端口号来区分不同的应用进程，从而实现真正的“端到端”通信。

### **1. “通信的端点并不是主机，而是主机中的应用进程”**

- 网络传输中，我们通常说两台主机之间通信，但**实际上通信的真正参与者是主机上的具体应用程序（进程）**。
- 比如：
  - 你用浏览器访问网页，这是一个浏览器进程；
  - 你用微信发送语音，这是另一个微信进程。

**所以通信不是"电脑对电脑"，而是"进程对进程"。**

------

### **2. “运输层通过端口号来区分不同的应用进程”**

- 每个应用进程通过不同的端口号与网络通信。
- **端口号就像门牌号**，帮助运输层（如TCP、UDP）把数据准确送到对应进程。
  - 例如 HTTP 默认用端口 80；
  - HTTPS 用 443；
  - FTP 用 21；
  - 本地运行的多个服务会绑定不同端口，比如本地调试常用 8080、3000 等。

------

### **3. “实现真正的‘端到端’通信”**

- 所谓“端到端”，**指的是进程到进程的通信，而不仅仅是主机到主机**。
- 一个完整的通信“端点”是：`IP地址 + 端口号`，即所谓的“套接字（socket）”。
- 比如：
  - A 主机上浏览器进程（端口12345）←→ B 主机上的 Web 服务器进程（端口80）
  - 这两个“端点”才是通信的真正两端。

------

### ✅ 总结一句话：

> **运输层通过“IP + 端口号”识别通信双方的具体应用进程，实现了“进程到进程”的真正端到端通信。**

  3. **动态或临时端口号（Dynamic/Private ports）：49152~65535**
     - 客户端临时使用，系统自动分配

------

### ✅ D. **熟知端口号 0~1023 专属服务器程序**

HTTP/1.0协议中的持续连接（也称为持久连接或长连接）允许在一个TCP连接上发送和接收多个请求和响应。这意味着，一个Web页面中的多个资源，如HTML文件和JPEG图形文件，可以通过同一个TCP连接来传送，而不是为每个资源单独建立一个连接。这样做的好处是减少了建立和关闭连接的开销，从而提高了传输效率。

具体来说，Web服务器在接收到用户机的请求后，会保持该TCP连接打开，直到服务器发送完响应或者连接超时。这样，客户机和服务器可以在同一个连接上进行多次请求和响应的交换。

所以，选项A错误地描述了HTTP/1.0中的持续连接行为。实际上，多个Web页的资源可以通过一个持续的TCP连接来传送，而不是必须使用多个连接

# 104.DHCP

![image-20250624152523072](./%E8%AE%A1%E7%BD%91.assets/image-20250624152523072.png)

**DHCP（Dynamic Host Configuration Protocol，动态主机配置协议）** 是计算机网络中**自动分配IP地址的核心协议**。它允许设备（如电脑、手机）自动获取网络配置参数，无需手动设置。以下是深度解析：

### **一、DHCP的核心作用**

| **手动配置的痛点**            | **DHCP的解决方案**                  |
| ----------------------------- | ----------------------------------- |
| 每个设备需手动设置IP地址      | **自动分配IP**（避免地址冲突）      |
| 需单独配置子网掩码、网关、DNS | **一键获取全套网络参数**            |
| 设备移动时需重新配置          | **动态适应不同网络**（如Wi-Fi切换） |

在DHCP客户与服务器交互过程中，描述错误的是选项B。以下是对每个选项的具体分析：

A. **正确**：租用应答报文DHCPOFFER确实包括分配给客户的IP地址、租用期与参数。这是DHCP服务器对客户端DHCPDISCOVER请求的响应。
B. **错误**：客户端租用请求报文DHCPDISCOVER并不是以点-点方式发送出去的，而是以广播方式发送。这是因为客户端在初始阶段并不知道DHCP服务器的IP地址，因此需要通过广播来寻找可用的DHCP服务器。
C. **正确**：接收到客户端请求报文DHCPDISCOVER的服务器都会返回一个应答报文DHCPOFFER，以提供可用的IP地址和其他配置信息。
D. **正确**：在客户端选择了一个DHCPOFFER之后，它会向选定的服务器发送DHCPREQUEST报文进行确认，服务器则回应以DHCPACK报文，完成IP地址的分配和租约的确认。

综上所述，选项B描述是错误的，因为它错误地描述了DHCPDISCOVER报文的发送方式。正确的应该是DHCPDISCOVER以广播方式发送，而不是点-点方式。

B. **注册认证RA中心确认申请下载数字签名密钥用户的合法身份**：这是正确的。RA（注册认证）中心负责验证用户的身份，确保只有合法用户才能获得数字证书和密钥对。

C. **认证中心CA产生用户之间通信时使用的非对称加密的公钥与私钥**：这是正确的。CA负责生成和管理公钥和私钥对，并将公钥与用户身份信息绑定在数字证书中。

D. **接收文件的用户用公钥验证文件的合法性**：这是正确的。在数字签名过程中，发送方使用其私钥对文件或消息进行签名，接收方则使用发送方的公钥来验证签名的合法性，从而确保消息的完整性和真实性。

# 110.IPSec

![image-20250624154803880](./%E8%AE%A1%E7%BD%91.assets/image-20250624154803880.png)

IPSec（Internet Protocol Security）是一种用于网络层安全的协议套件，它提供数据源认证、数据完整性、数据保密性以及防重放攻击等服务。主要特征包括：

A. IPSec的安全服务工作在网络层。这意味着它可以在OSI模型的网络层对数据进行保护。

B. IPSec是由多种安全协议与加密算法构成的一个安全体系。它包括AH（Authentication Header）和ESP（Encapsulating Security Payload）两种主要协议，以及各种加密和认证算法。

C. 这个选项是错误的。IPSec对于IPv4是可选的，但对于IPv6是强制实施的。IPv6协议栈要求必须支持IPSec。

D. IPSec定义了两种保护IP分组的模式，分别为传输模式与隧道模式。传输模式通常用于保护端到端通信，而隧道模式则用于保护整个IP包，常用于VPN场景。

因此，关于IPSec主要特征的描述中，C选项是错误的。

# 111.防火墙

![image-20250624154917045](./%E8%AE%A1%E7%BD%91.assets/image-20250624154917045.png)

防火墙系统是一种网络安全设备，用于监控和过滤进出网络的数据流，以保护网络免受未经授权的访问和攻击。关于防火墙系统的描述，正确的有：

A. 检查所有从内部流出到外部网络的数据包：防火墙确实会检查所有从内部网络流向外部网络的数据包，以确保这些数据包的合法性，并阻止潜在的安全威胁。

B. 对于不同协议的分组进行格式变换：防火墙并不主要负责对不同协议的分组进行格式变换。这通常是网络层或其他网络设备的功能。

C. 包过滤路由器按照分组过滤规则，只需要根据IP地址决定该分组是否应该转发：包过滤防火墙确实会根据预设的规则过滤数据包，这些规则可以基于IP地址、端口号、协议类型等。但“只需要根据IP地址决定该分组是否应该转发”过于简化了包过滤路由器的功能。

D. 最简单的防火墙由一个应用级网关组成：虽然应用级网关是防火墙的一种类型，但并不是所有防火墙都由应用级网关组成。防火墙可以有多种实现方式，包括包过滤防火墙、状态检测防火墙等。

综上所述，选项A描述是正确的。防火墙会检查所有从内部流出到外部网络的数据包，以确保网络安全。
